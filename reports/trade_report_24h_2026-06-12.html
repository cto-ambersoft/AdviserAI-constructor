⚠️ Где сервис считает неправильно (UI показывает 274 как убыток вместо прибыли)

Баг 1 — реализованный PnL позиции = 0.
build_position_pnl_snapshot → _position_ledger_breakdown (service.py:2546 / 5269) берёт филлы по auto_trade_position_id == position.id. Но все выходные филлы (TP/SL) приходят с origin='external' и auto_trade_position_id = NULL — user-data WS их пропустил, а REST-синк не привязал к позиции. К 274 «привязаны» только 6 входных филлов (realized_pnl=0). Сервис выдаёт gross_realized = 0, net = −0.22 USDT и показывает позицию как мелкий минус, тогда как по бирже это +2.98 прибыли. На реальном счёте это системно: практически все PnL-несущие филлы — external и не привязаны ни к одной позиции.

Баг 2 — фандинг позиции = 0 (несовпадение формата символа).
sum_funding (income_sync.py) фильтрует income_ledger.symbol == position.symbol, где position.symbol = "BTC/USDT:USDT" (ccxt-формат), а реестр доходов хранит символ как "BTCUSDT" (нативный Binance). Совпадений нет → фандинг по позиции всегда 0. Реальный фандинг 274 = −0.0095 USDT.

Числа в этом отчёте взяты не из багового сервисного метода, а посчитаны напрямую из биржевого реестра по связке символ + время и сверены с Binance — поэтому корректны. Баги бьют по отображению PnL позиции в UI/API. Отдельно стоит проверить, не ключуются ли так же риск-гард / дневной лимит убытка (если да — риск-решения могут опираться на заниженный PnL).

Методология и примечания

Область отчёта: только реальный счёт Binance (account 17). Демо-счёт (account 16) исключён по запросу.

Источники данных (всё read-only): позиция — auto_trade_positions; филлы, комиссии, реализованный PnL — exchange_trade_ledger (realized_pnl, fee_cost); фандинг — exchange_income_ledger (income_type='FUNDING_FEE'). Дополнительно — прямой read-only запрос к Binance live fapi для сверки.

Реализованный PnL — сумма realized_pnl по закрывающим филлам (брутто, до комиссии). Комиссия — сумма taker-fee по всем филлам (вход + выходы). Фандинг — по Binance income со знаком: отрицательный = уплачено, положительный = получено; начисляется каждые 8 ч (00:00 / 08:00 / 16:00 UTC) на открытую позицию. Нетто = realized − commission − funding.

«external» филлы: TP/SL-исполнения по 274 пришли из биржи мимо платформенного учёта (user-data WebSocket их пропустил) и доведены REST-синком. Реализованный PnL и комиссия по ним взяты из биржевого реестра и учтены (см. раздел «ошибки учёта» — сервис их к позиции не привязывает).

Открытый остаток (274): «нетто» — только по закрытым 0.004 BTC. Плавающий PnL по остатку 0.003 BTC дан live-значением Binance по mark-цене и не зафиксирован.

Контекст по 274: первая позиция реального счёта после деплоя фикса переноса SL при срабатывании TP на «глухом» WS. Перенос SL отработал (REST-контур), позиция всё время оставалась под защитой стоп-лосса.
